## 1. 먼저 알아야 할 개념

### 1.1 재생 "엔진"이 왜 필요한가
지금까지 만든 `speakAndAwait`는 문장 **한 개**를 읽고 기다리는 조각이었다. 하지만 앱은 클립(문장 수십~100개)을 **설계된 흐름대로 쭉** 재생하고, 그 중간에 사용자가 앞/뒤로 움직일 수 있어야 한다. 그 "쭉 재생 + 이동"을 담당하는 게 **시퀀서(sequencer)** 다.

### 1.2 StateFlow = "구독하는 현재 상태"
`StateFlow`는 값이 바뀔 때마다 **구독자에게 자동으로 알려주는 상자**다. 시퀀서가 "지금 3번 문장, 일본어 읽는 중"을 StateFlow에 담으면, 화면은 그걸 구독해 자동으로 다시 그린다. (백엔드의 상태 푸시/구독과 같은 개념)

### 1.3 Service / Foreground Service
- **Service**: 화면 없이 뒤에서 도는 작업 단위.
- **Foreground Service**: "지금 사용자가 인지하는 중요한 작업 중"이라고 OS에 알리는 서비스. **알림을 반드시 띄워야** 하고, 대신 OS가 함부로 안 죽인다. 음악 앱이 이걸로 백그라운드 재생을 한다.
- **바인딩(bind)**: 화면이 서비스에 "연결"해서 상태를 구독하고 명령을 보내는 것. 화면이 죽어도 서비스는 살아있다.

### 1.4 Compose UI = f(state)
Compose는 "상태를 넣으면 화면이 나오는 함수"다. 위젯을 직접 조작하지 않고, **상태만 바꾸면 화면이 알아서 다시 그려진다.** 그래서 재생 상태(StateFlow)를 구독만 하면 UI가 저절로 따라온다.

### 1.5 직렬화(serialization)
문자열(JSON) ↔ 코틀린 객체를 자동 변환하는 것. `kotlinx.serialization` 라이브러리 + 모델에 `@Serializable`을 붙이면, `assets/clips/n4_office.json` 같은 파일을 `Clip` 객체로 바로 읽어들일 수 있다.

## 2. 재생 엔진 — `TtsSequencer` (TODO 2)

### 2.1 핵심 아이디어 — "평탄화"
문장을 그대로 다루지 않고, **JP/KR/단어를 전부 똑같은 `PlaybackStep`(스텝)으로 펼친다.**

```
문장1 → [JP, JP, JP, KR, JP, 단어JP, 단어KR], 문장2 → [...], ...
```

클립 전체를 **하나의 긴 스텝 리스트**로 만들면, 재생 엔진은 그저 이 리스트를 **인덱스로 순회하는 루프** 하나면 된다:

```kotlin
while (currentStepIndex < steps.size) {
    val step = steps[currentStepIndex]
    tts.speakAndAwait(step.text)   // 다 읽을 때까지 대기
    if (step.pauseAfterMs > 0) delay(step.pauseAfterMs)  // 정지
    currentStepIndex++
}
```

이 단순 루프에서 모든 기능이 공짜로 나온다:
| 기능 | 구현 |
|---|---|
| 연속 재생 | 루프가 끝까지 도는 것 = 기본 |
| 다음/이전/다시듣기 | **인덱스 점프** (루프 취소 → 인덱스 이동 → 재개) |
| 일시정지 | 코루틴(루프) 취소 + `tts.stop()` |
| 속도 | 값만 바꾸면 다음 스텝부터 반영 |

### 2.2 파일
- `Playback.kt` — 데이터 모델 + 스텝 평탄화(`toSteps`) + 재생 패턴 프리셋
- `TtsSequencer.kt` — 루프 + 현재 인덱스 + `StateFlow<PlayerUiState>` + 조작 메서드

> 상태는 `state`(StateFlow)로만 노출하고, 조작은 메서드로만 받는다 = **단방향**. UI는 구독+명령만.

## 4. 재생 흐름 다듬기 (실제로 들어보며 조정)

실기기로 들으며 흐름을 다음처럼 잡았다. 값은 전부 `PlaybackPattern`이라 나중에 설정/문장별로 바꿀 수 있다.

**DRILL(문장 드릴) 한 문장 흐름:**
```
JP① →1.5초→ JP② →1.5초→ JP③ →3초(해석)→ KR →0.8초→ JP④(뜻 알고 다시) →1.5초→ 단어(일→한) →2초(문장 사이)→ 다음 문장
```
그리고 **클립 끝**에 종료 안내: 「聞き取りが終わりました」 → 「듣기가 끝났습니다」.

조정 내역(요청 반영):
- **일본어 3회가 너무 빨라서** → 회차 사이 1.5초 정지 추가(`pauseBetweenRepeatsMs`).
- **쉼표(、)를 그냥 이어 읽어서** → 쉼표에서 끊고 0.4초 무음 삽입(`speakAndAwait`가 문장을 쉼표로 쪼개 재생).
- **흐름 변경 요청** → `JP×3 → KR → JP×1 → 단어` (뜻을 들은 뒤 다시 일본어 한 번).
- **문장 사이가 너무 빨라서** → 문장 사이 2초 정지(`pauseBetweenSentencesMs`).

## 5. 백그라운드 재생 — `PlaybackService` (TODO 4)

### 5.1 왜 필요했나 (함정)
검증 중 "화면 닫아도 재생되던데?" 했지만, 그건 **액티비티가 아직 안 죽어서** 잠깐 되던 것. 포그라운드 서비스 없이 백그라운드에 있으면 안드로이드(특히 삼성)가 **예고 없이 앱을 재워서** 소리가 끊긴다. 그래서 재생을 **서비스가 소유**해야 한다.

### 5.2 구조
```
PlaybackService (Foreground, type=mediaPlayback)
 ├── TextToSpeech + TtsSequencer  ← 재생을 여기서 소유(화면 아님)
 ├── 미디어 알림                   ← 이전/재생·일시정지/다음/정지 버튼
 ├── MediaSession                 ← 잠금화면·이어폰 미디어 버튼
 └── AudioFocus                   ← 전화·타앱 소리 시 일시정지, 이어폰 뽑힘 시 정지
```

- 화면(`PlayerScreen`)은 서비스에 **바인딩**해서 상태(StateFlow)를 구독하고 명령만 보낸다.
- `startForeground` + 알림 → 화면 꺼도/앱 나가도 **프로세스가 안 죽음**.
- 재생 시작 시 `startService`로 "started 서비스" 승격 → 화면이 언바인드(나가도)해도 재생 유지.

### 5.3 개념 — 왜 서비스가 재생을 "소유"하나
재생 상태와 루프를 화면이 들고 있으면 화면이 죽을 때 재생도 죽는다. **서비스가 소유**해야 화면과 무관하게 소리가 이어진다. 화면은 "붙어서 구경하고 명령하는" 얇은 존재.

### 5.4 결과
실기기(갤럭시 S24+)에서 화면 끄기·앱 나가기·잠금화면·이어폰·전화 인터럽트 전부 확인 통과.

## 7. 콘텐츠 — N4 300문장 (병렬 생성)

카테고리당 100문장, 총 300문장을 손으로 쓰면 오래 걸려서 **병렬 에이전트 3개**에게 동시에 생성시켰다(회사/여행/생활회화 각 1개). 각 에이전트가 JSON 파일을 직접 작성.

- 각 문장: 일본어 + 한국어 + 핵심 단어 2~4개. N4 문법(て형/た형/～たい/～なければならない/가능형/수수동사/～たことがある/～つもり/～かもしれない 등) 폭넓게, 상황 다양하게.
- 생성 후 **JSON 유효성 자동 검증**(정확히 100문장, id 연속, 파싱 성공)으로 3파일 모두 확인.
- 어색한 번역 1곳만 수정("맑는다고" → "맑을 거라고").

> 주의: 300문장을 자동 생성한 것이라 어색한 표현이 더 있을 수 있다. 쓰다가 발견하면 그때그때 수정.

### 7.1 랜덤 클립
`AssetClipRepository.randomClip()` — 모든 클립 문장을 합쳐 **섞은 뒤 30개**를 뽑아 임시 클립으로 만든다(id 0). 다른 클립에서 랜덤을 열 때마다 새로 섞인다. 목록의 🔀 카드가 이걸 연다.

## 8. 음질 개선 — 목소리 갈라짐 (요청 반영)

"가끔 목소리가 갈라진다"는 피드백. 원인은 ① 내장 TTS 엔진 특성(원래 좀 그럼) + ② **우리 코드가 매 발화마다 엔진을 재설정**하던 것.

- 기존: 스텝마다 `setSpeechRate` + `setPitch` + `language`를 매번 다시 설정 → 이음새가 거칠어짐.
- 개선: 시퀀서가 **속도·언어가 실제로 바뀔 때만** 설정하고, 불필요한 pitch 리셋 제거. (JP×3 같은 연속 구간은 재설정 0회)

근본 해결은 추후 §10.3(ElevenLabs mp3)이지만, MVP 내장 TTS에서 할 수 있는 개선은 적용.
