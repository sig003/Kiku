## 1. 이어폰/블루투스 미디어 버튼 (가장 까다로웠음)

### 1.1 증상
유튜브는 이어폰 버튼으로 정지되는데 우리 앱은 안 됨.

### 1.2 개념 — "미디어 버튼 세션"
안드로이드에서 이어폰/BT의 재생·정지 버튼은 **현재 "미디어 버튼 세션"으로 지정된 앱 하나**에게만 간다. 그 세션은 **실제로 오디오를 재생 중인 앱**에게 주어진다.

### 1.3 진짜 원인 (dumpsys로 확인)
`adb shell dumpsys media_session` 로 보니, 우리 앱이 **재생 중(PLAYING)이고 세션도 active인데도 "Media button session is null"**. 경쟁 앱(유튜브 뮤직)을 꺼도 여전히 null.
→ 원인: **우리 오디오를 TTS 엔진(별도 프로세스)이 내보내기 때문.** OS의 오디오 모니터는 "com.bradlab.kiku가 오디오를 재생 중"이라고 인식하지 못한다(소리는 TTS 엔진 프로세스가 냄). 그래서 우리 세션이 미디어 버튼 후보에서 빠진다.

### 1.4 해결 — 무음 킵얼라이브
재생 중 **우리 프로세스에서 무음 `AudioTrack`을 계속 흘린다.** 그러면 OS가 "이 앱이 오디오 재생 중"으로 인식 → 우리 세션이 미디어 버튼 세션이 됨.
- `startKeepAlive()`(무음 8kHz mono PCM 루프) / `stopKeepAlive()` — play/pause/stop/destroy에 연결.
- 추가로 `MediaSession.setMediaButtonReceiver(...)` + `setFlags(...)`, `onStartCommand`에서 `ACTION_MEDIA_BUTTON` 처리, `onMediaButtonEvent`로 키코드 직접 처리.
- 검증: dumpsys에서 `Media button session=kiku`, adb 미디어키(85/87)로 재생·정지·다음 동작 확인 → 실기기 이어폰으로도 OK.

## 2. 단어(어휘) 정제

### 2.1 두 가지 문제
- N3 단어 뜻에 **'(수동)/(사역수동)/(피해수동)'** 문법 태그가 붙어 TTS가 "…물결 합니다" 식으로 읽어버림 → 태그 20곳 제거(뜻은 유지).
- 문장마다 **너무 쉬운 단어(오늘/나/이것…)** 까지 추출 → 허탈. **레벨에 맞는 가치 있는 단어만** 남김.

### 2.2 방법
- 파일별 **병렬 에이전트**로 `words`만 정제(문장·번역·순서 불변). N4는 N5어 제거, N3는 N4 이하까지 제거.
- 전수 검증: 100문장 유지·`jp`/`kr` 원본과 동일·JSON 유효. (에이전트가 6개 동시엔 API 오류로 실패 → 하나씩 재실행해 완료)
- 결과: n4 각 233→127 수준, n3 각 →145~198.

## 3. 대화(DIALOGUE) 모드

### 3.1 무엇
한 문장씩 **A(질문) ↔ B(답변)** 주고받는 대화. 콘텐츠에 `speaker`("A"/"B") 지정, `mode: DIALOGUE`.

### 3.2 재생 패턴 — "세트로 주고받기"
처음엔 A를 3번, B를 3번 드릴했더니 대화가 아니었다. 요청대로 **(A + B)를 한 세트로**:
```
[A일본어, B일본어] ×3  →  [A한국어, B한국어]  →  [A일본어, B일본어] ×1  →  단어(A,B)  →  다음 짝
```
`Clip.toSteps()`가 `mode == DIALOGUE`면 `dialogueSteps()`(두 문장씩 짝지어 평탄화), 아니면 기존 `drillSteps()`.

### 3.3 콘텐츠
- `n4_dialogue.json`(id 4, N4, 50문장/6상황), `n3_dialogue.json`(id 14, N3, 50문장). 화자 A/B 교대.
- 목록 아트 타일에 **대화=2명 배지**(`PeopleGlyph`, 벡터로 두 사람) + 카테고리별 `対話`(보라) 아트.

## 4. 화자 목소리 (남/여 매핑)

### 4.1 화자별 목소리
`TtsSequencer`가 언어별로 **성별 다른 목소리 2개**를 골라, 대화의 화자에 매핑. `speakerOrder`(A/B 등장 순서)로 A=슬롯0, B=슬롯1.

### 4.2 조정 내역(피드백 반영)
- **항상 여자가 먼저** → **짝(exchange)마다 A/B 남녀 순서를 무작위**(`pairVoiceSwap`, 클립 열 때 한 번 정함). 남→여·여→남 섞임.
- **한국어 해석 목소리 정리** → 처음엔 한 목소리로 통일했다가, 요청대로 **일본어와 같은 슬롯(=같은 성별)** 으로. 단어(어휘)만 **한 목소리(나레이터, 슬롯0)** 로 통일.
- **한국어가 둘 다 여성으로 나옴** → 기기 한국어 목소리 4종(ism/kob/koc/kod) 중 자동 선택된 2개(ism/kob)가 **둘 다 여성**이라 남자가 안 나온 것. → **알려진 남성 코드(ko: koc/kod, ja: jac/jad)** 로 `[여성, 남성]` 순서를 맞춰 선택. 이러면 일/한 **슬롯 성별이 정렬**돼 A의 일본어·한국어가 같은 성별로 나옴.

### 4.3 개념 — 목소리 성별 판별의 한계
안드로이드 `Voice`는 **성별을 직접 알려주는 API가 없다.** 그래서 구글 온디바이스 목소리 코드(경험적으로 알려진 남성 코드)로 힌트를 준다. 기기/엔진이 다르면 어긋날 수 있어, 폴백(이름순 2개)을 둔다.
> 정확한 목소리 통제가 필요하면 결국 §10.3(ElevenLabs mp3 사전 생성)로 가야 한다 — 내장 TTS는 기기 편차가 큼.
