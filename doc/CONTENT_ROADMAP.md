# kiku 콘텐츠·기능 로드맵

> 앞으로 "무엇을 만들지"의 기준 문서. 2026-07-12 논의 정리.
> 세 관점을 그대로 남긴다: ① 초안(콘텐츠 목록) → ② 사용자 관점 재정렬(왜 매일 켜나) → ③ 실행 분석(기능 vs 콘텐츠). 결론은 맨 끝 **최종 실행 순서**.
> 현재 타깃: **N4·N3 학습자**(가장 두터운 층). 전략은 "N3까지를 가장 잘 만든 앱으로" → 그 뒤 N2 확장.

## ① 초안 — 콘텐츠 중심 목록 (첫 제안)

> "무슨 콘텐츠를 넣을까" 관점. 아래 ②③에서 보강·재정렬됨.

**N4/N3 추가 후보**
1. **실전 청해 유형(LISTENING)** ⭐ — `LISTENING` 모드가 비어 있음. JLPT 청해 실제 유형을 채우면 "연습앱 → 시험대비앱"으로 격상.
   - 課題理解(과제이해): 대화 듣고 "다음에 뭘 해야 하나"
   - ポイント理解(요점이해): "왜/무엇을" 집어내기
   - 概要理解(개요이해): 안내방송·모놀로그의 요지
2. **주제 카테고리 확장** — 병원·약국 / 쇼핑·계산 / 전화 / 식당·예약 / 학교·수업 / 은행·우체국·관공서 / 부동산 / 교통·길찾기
3. **기능·문법 특화 클립** — 경어(존경↔겸양) / 조수사(個·枚·本·杯) / 시간·날짜·요일 / 의성어·의태어 / "N4 핵심문법 50"·"N3 핵심문법 50"

**N2/N1**
- 앱 강점은 "짧은 문장 반복 청취" → 상위급은 청해가 길고 추상적이라 단문 드릴 가치가 하락.
- **N2**: 뉴스·사회 짧은 문장, 비즈니스 회화, 관용구·사자성어, 조금 긴 대화(DIALOGUE↑) → 낼 만함(수요 있음).
- **N1**: 강의체·논설·뉴스·격식·장문 → 제작 난이도 높고 수요 적음 → 보류.
- 핵심: **상위급일수록 단문보다 대화·청해(맥락) 비중을 키운다.**

## ② 사용자 관점 재정렬 — "왜 이 앱을 계속 켜게 될까"

> "무슨 콘텐츠"보다 **왜 매일 켜나**를 먼저 생각하면 우선순위가 바뀐다.

**1순위. JLPT 실전 청해 ⭐⭐⭐⭐⭐** — 무조건.
시험은 문장만 듣는 게 아니다: 안내방송·역 방송·회사 대화·전화·가게·학교·공지·일정 변경 형태가 훨씬 많다.
```
男：今日は会議室ではなく、三階の研修室です。
↓ 문제: どこへ行きますか。
```
실전 감각을 키우는 콘텐츠는 경쟁력이 크다.

**2순위. 쉐도잉(따라 말하기) ⭐⭐⭐⭐⭐**
지금은 "듣기"에서 끝. `듣기 → 따라 말하기 → 다시 듣기`까지 가면 기억이 오래간다.
```
문장 재생 → 3초 정적 → 사용자가 따라 읽음 → 다시 원어민 음성
```

**3순위. 문법별 청해 ⭐⭐⭐⭐**
예) N3 〜ことになる 100문장 / 〜ようにする 50 / 〜わけではない 50.
문법이 귀에 안 들어오는 사람이 많다 → 수요 있음.

**4순위. 헷갈리는 단어 모음 ⭐⭐⭐⭐**
早い vs 速い / 温かい vs 暖かい / 帰る vs 戻る / 聞く vs 聞こえる 를 문장으로 계속 들려주기. (블로그 노트 자산과 연결)

**5순위. 속도별 재생 ⭐⭐⭐⭐**
0.8 / 1.0 / 1.2배. N3 이후 속도 적응이 중요. *(주: 이미 구현됨 — ③ 참고)*

**6순위. 상황별 세트 ⭐⭐⭐⭐**
병원·미용실·편의점·식당·은행·우체국·공항·호텔·시청(관공서)·재난방송·전화·택배. JLPT에도 자주 나온다.

**7순위. 일본인이 자주 쓰는 표현 ⭐⭐⭐**
なるほど / そうなんですね / やっぱり / とりあえず / ちなみに / まさか. 애니·드라마에도 자주.

**N2부터는 방향 전환** — 단문보다 맥락: 뉴스·회의·전화 응대·프레젠테이션·인터뷰·라디오·강의·의견 교환.

**추가 제안: 「오늘의 10문장」** — 매일 10문장 자동 추천. "오늘은 이 10개만 듣자"는 목표가 생겨 **매일 켜게 된다.**

**사용자가 매긴 개발 우선순위**
1. JLPT 실전 청해(최우선) 2. 상황별 확장 3. 쉐도잉 4. 문법별 청해 5. 헷갈리는 단어 6. 매일 추천(오늘의 10문장) 7. N2(뉴스·비즈니스·장문) 8. N1

**전략 결론(사용자):** 너무 빨리 N2/N1로 넓히기보다 **N3까지를 '가장 잘 만든 앱'으로.** 타깃(N4·N3)이 가장 두텁고, 이 구간의 품질·반복 경험을 확실히 하면 N2 추가 시 기존 사용자가 자연스럽게 이어 쓴다.

## ③ 실행 분석 — "기능 vs 콘텐츠" 렌즈 (개발 효율)

> 방향은 ②에 동의. 개발자 관점에서 **레버리지**로 우선순위를 다듬는다.

### 렌즈 1: 성격이 두 종류다
| 성격 | 항목 | 특징 |
|---|---|---|
| **기능(코드)** | 쉐도잉, 속도별, 오늘의 10문장, 스트릭 | 한 번 만들면 **모든 레벨·모든 클립에 자동 적용**. 콘텐츠 0개 추가로 앱이 좋아짐(레버리지 큼) |
| **콘텐츠(데이터)** | 실전 청해, 상황별, 문법별, 헷갈리는 단어, 표현집, N2/N1 | 만든 만큼만 늘어남(선형). 계속 찍어내야 함(무거움) |

→ "적은 노력 큰 효과" 순서는 **기능 먼저**.

### 렌즈 2: 이미 됐거나 거의 된 것 (중복 방지)
- **속도별 재생**: 이미 구현 ✅ (5순위는 완료 항목)
- **쉐도잉**: `PlaybackPattern.pauseAfterJpMs`(문장 뒤 정적) 이미 있음 → **쉐도잉 프리셋 하나 추가**면 끝. 새 엔진 불필요.
- **오늘의 10문장**: `randomClip`(레벨·셔플) + 진행 저장 이미 있음 → **"10개 뽑아 오늘 고정 + 스트릭"만 얹으면 됨.**

### 렌즈 3: 자기 논리와의 정합성
- "왜 매일 켜나 → 오늘의 10문장"이라 했으면, 그건 **6위가 아니라 앞쪽**이어야 한다.
- **실전 청해(1순위)는 가장 강력하지만 가장 무거운 콘텐츠**(문항마다 대화+문제+정답+오답3). 이것만 붙들면 몇 주간 "매일 켤 이유"가 안 생긴다.
- 리텐션은 콘텐츠 양이 아니라 **습관 장치**에서 나온다 → **오늘의 10문장 + 연속 학습일(스트릭)** = 초저비용·즉효.

## ④ 최종 실행 순서 (결론)

**Phase 1 — 리텐션 훅 먼저 (기능, 각 1~2일, 콘텐츠 0)**
1. **오늘의 10문장 + 스트릭(연속 학습일)** — "매일 켜는 이유" 직접 해결. 기존 랜덤/진행저장 재활용.
2. **쉐도잉 프리셋** — 문장 뒤 정적 이미 있음, 프리셋만 추가.

**Phase 2 — 차별화 콘텐츠 (실전, 병행 착수)**
3. **실전 청해(LISTENING: 과제이해·요점이해·개요이해)** — QUIZ 인프라 재활용해 유형 확장.
4. **상황별 확장** — 병원·전화·쇼핑·관공서·재난방송·공항·호텔 등.

**Phase 3 — 콘텐츠 시리즈 (블로그 자산 연계)**
5. **헷갈리는 단어**(早い/速い…) — 블로그 노트가 소스, 소싱 우위.
6. **문법별 청해**(〜ことになる 100문장 등).
7. **자주 쓰는 표현**(なるほど·とりあえず…).

**Phase 4 — 상향 확장**
8. **N2**(대화·뉴스·비즈니스 위주, DIALOGUE/LISTENING 비중↑).
9. **N1**(보류 — N2 안착 후).

**전략(공통):** 좁고 깊게(N3까지 완성) → 그다음 넓게(N2). Phase 1 기능을 N3까지 완성해두면, 이후 N2 콘텐츠를 부어도 그 기능이 공짜로 적용된다.

## ⑥ 음성 입력(STT) — 쉐도잉 확장 검토 (2026-07-12)

> "사용자 음성을 받아 처리할 수 있나?" 논의 정리. 쉐도잉(Phase 1-2)의 **선택적 확장**으로 둔다.

### 기술 옵션 3가지
| 방식 | 하는 일 | 비용/조건 | 이 앱에 |
|---|---|---|---|
| **A. Android `SpeechRecognizer`(내장)** | 사용자가 일본어로 말함 → 텍스트로 변환(STT) → 원문과 문자열 비교로 "맞았다/틀렸다" | 무료, `RECORD_AUDIO` 권한, 대체로 인터넷 필요 | ✅ 쉐도잉 확인용 적당 |
| **B. 발음 평가 API**(Azure Speech 등) | 발음 점수(정확도·유창성·완성도) + 틀린 음절 표시 | 유료(클라우드), 인터넷 필수, 서버/키 관리 | ⚠️ 진짜 발음교정엔 최고지만 MVP 밖 |
| **C. 오디오 녹음만** | 발화를 녹음 → 본인이 듣고 비교 | 무료, 인식 안 함 | 🔵 가장 단순, STT 오인식 회피 |

### 트레이드오프 (도입 전 반드시 고려)
1. **정체성 충돌** — kiku는 "화면 안 보고 듣기만"(라디오형)이 핵심. 음성 입력은 화면 보며 하는 능동 학습이라 **별도 모드로 분리**해야 함.
2. **권한·스토어 선언** — 지금은 권한 최소("데이터 수집 없음")가 강점. `RECORD_AUDIO`+STT(인터넷)가 붙으면 **Play 데이터 안전 선언이 복잡**해짐(마이크·네트워크). 단순함을 깰 값어치가 있는지 따질 것.
3. **STT 정확도** — 학습자의 서툰 발음은 오인식이 잦아 맞는데도 "틀렸다"고 오판 → 학습 의욕 저하 위험.

### 추천 순서
1. **쉐도잉 먼저 — STT 없이**(= Phase 1-2): `문장 → 정적 → 다시 듣기`. 정적 이미 있어 거의 공짜, 정체성 안 깸. 학습 효과 대부분 여기서.
2. **원하면 그다음 A(내장 STT)**: 쉐도잉에 "확인" 버튼 → 말하면 텍스트로 받아 원문 대조("맞췄나" 수준). 무료.
3. **발음 점수(B)는 한참 뒤**: 유료·서버 필요, 수요 검증 후.

**결론:** 가능하고 내장 기능(무료)으로 STT까지 붙일 수 있음. 단 "듣기 전용+권한 최소" 강점을 지키려면 **쉐도잉은 STT 없이 시작, 음성 입력은 선택 기능으로 나중에** 얹는다.

## ⑦ 방향 전환 — "공부 앱"이 아니라 "습관 앱" (2026-07-12)

> UX 톤 자체를 다시 잡는 논의. 콘텐츠보다 **경쟁 축**을 바꾸는 이야기라 가장 상위 전략에 가깝다.

### 문제의식 (사용자 제기)
- 지금 앱이 **딱딱하고 "공부해야 한다"는 압박·무거운 느낌**이 큼.
- JLPT 앱들이 실패하는 이유 중 하나 = 켜는 순간 "문제집을 펼친 느낌". 진입 장벽이 심리적임.
- 매일 들어오는 앱들은 **공부 앱이 아니라 습관 앱**처럼 느껴진다.
- KIKU의 핵심(걸으면서 듣기·한 문장 반복·부담 적음)은 이미 잘 잡혔는데, 화면·콘텐츠가 "공부"에만 초점이라 장벽이 생김.

### 넣고 싶은 것 (사용자 리스트)
1. **오늘의 한 문장(최우선)** — 켜면 맨 위에 딱 하나. 누르면 15초. "오늘 이것 하나만 듣고 가자" → **공부가 아니라 출석 느낌.**
2. **연속 기록(Streak)** — 🔥 7일 연속. 사람은 공부보다 **연속 기록 끊기를 싫어함.**
3. **랜덤 버튼** — 🎲 아무거나 하나. 뭘 공부할지 고민 안 하게.
4. **오늘의 표현** — 今日は暑いですね。/ 오늘 덥네요. 매일 하나씩, SNS 짧은 콘텐츠처럼.
5. **"3분이면 끝"** — 버튼 이름: `공부 시작 ❌` → `3분 듣기 ⭕`. 훨씬 가볍게.
6. **일본 생활 잡지 느낌** — 🍣 편의점에서 많이 쓰는 표현 / 🚃 전철에서 자주 듣는 방송 / ☔ 오늘의 일본어. 분위기가 살아남.
7. **귀여운 마스코트** — 정답/오답 대신 👂 "오늘도 귀가 열리고 있습니다" / 🎌 "한 문장 더?" 같은 가벼운 반응 캐릭터.

**사용자의 방향 제안:** "문제를 많이 푸는 앱"으로 경쟁하면 못 이김. KIKU 정체성을 살려 **"공부한다"보다 "매일 일본어를 듣는 습관"** 으로. 앱 전체 톤도 문제집보다 **음악/팟캐스트 앱처럼 편안하게.**

첫 화면 예시(부담 적은 버전):
```
👋 안녕하세요! 오늘은 딱 한 문장만 들어볼까요?
 ▶ 오늘의 한 문장 (20초)
 🎲 랜덤 듣기
 🚶 산책하면서 5분 듣기
 🔥 12일 연속 듣는 중
```
→ "공부하러 왔다"가 아니라 "잠깐 들으러 왔다"는 느낌.

### 판단 (동의)
- **강하게 동의.** 경쟁 축을 "문제 수 → 매일 켜는 경험"으로 옮기는 것 자체가 전략.
- 다른 JLPT 앱은 다 문제집이라 물량 싸움에선 개인이 못 이김. **"팟캐스트/음악 앱 같은 편안한 듣기 습관 앱"은 아직 비어 있음.**
- KIKU의 걸으면서 듣기·백그라운드·한 문장 반복이 이미 그 방향 → **UX 톤만 습관앱으로** 바꾸면 정체성이 뚜렷해짐.
- 이 방향은 로드맵 **Phase 1(리텐션 훅: 오늘의 10문장·스트릭)** 과 정확히 일치. 버튼 문구·마스코트·톤은 저비용으로 큰 인상 변화.

## ⑧ 오늘의 일본 (뉴스 기반 데일리 콘텐츠) — 킬러 아이디어 (2026-07-12)

> "일본 뉴스/SNS/유튜브의 오늘 핫이슈를 받아 일본어로 읽고 번역 + 주요 단어·문법을 붙이면?" 논의 정리.

### 왜 강력한가
- **시의성 = 매일 켤 이유.** 고정 문제집은 어제나 오늘이나 같지만 "오늘의 뉴스"는 매일 다름 → 리텐션의 정답. ⑦ 습관앱 방향과 완벽히 맞물림.
- **콘텐츠 제작 병목을 자동화로 해결.** 현재 최대 약점(JSON 손으로 찍기)을 파이프라인이 대신함.
- **차별화.** 다른 앱엔 없는 "살아있는 콘텐츠". "오늘의 일본"이 브랜드가 됨.

### 기술 파이프라인
```
① 소스 수집  →  ② LLM 가공  →  ③ TTS 재생  →  ④ 앱 배달
```
| 단계 | 방법 | 난이도 |
|---|---|---|
| ① 소스 | **NHK News Web Easy**(やさしい日本語=학습자용 쉬운 뉴스, RSS) ⭐ / 각사 뉴스 RSS | 쉬움 |
| ② 가공 | LLM이 뉴스 → 짧은 학습 문장 재구성 + 한국어 번역 + 주요 단어·문법 추출 → 기존 JSON 스키마(Clip/Sentence/Word) 그대로 | 쉬움 |
| ③ 음성 | 기존 TTS(일/한) 그대로 재생 | 이미 됨 |
| ④ 배달 | cron이 매일 JSON 생성 → 정적 호스팅 → 앱이 fetch | 중 |

- **핵심:** DESIGN.md에서 `ClipRepository`를 interface로 두고 "추후 `RemoteClipRepository` 교체"를 이미 설계함(§10) → **원격 콘텐츠 그릇이 이미 준비됨.** 구현체만 추가.
- **소스 추천:** NHK News Web Easy = 이미 학습자용으로 난이도 조정된 일본어 → N4/N3와 정확히 일치. 유튜브/X는 API 비용·저작권·크롤링 제약이 커서 초기엔 부적합.

### 리스크 3가지 (처음부터 못 박을 것)
1. **저작권** — 원문 그대로 읽기 ❌. LLM으로 **사실만 뽑아 새 문장으로 재구성**(사실엔 저작권 없음, 표현을 새로 쓰면 OK). NHK Easy 원문 복제·재배포 금지.
2. **자동 생성 품질** — LLM 오역·부자연 가능 → **초기엔 반자동**(생성 → 사람이 훑고 승인 → 배포). 시의성 콘텐츠는 오역 시 신뢰 타격 큼.
3. **서버 필요** — 지금 서버리스라 매일 갱신은 원격 필요. 단 **가볍게** 가능: GitHub Actions(cron) → JSON 생성 → 정적 호스팅(GitHub Pages/S3/Firebase, 거의 무료) → 앱 fetch. 상시 서버 불필요.

### 단계적 실행
- **1단계(반자동, 인프라 0):** 매일 NHK Easy 1건 → LLM 생성 → 검수 → git commit → 앱 번들. "오늘의 뉴스"를 수동 배포로 먼저 검증.
- **2단계(자동 파이프라인):** cron 생성 → 정적 호스팅 → `RemoteClipRepository`로 fetch. 진짜 "매일 자동".
- **3단계:** 유튜브/SNS는 저작권·API 정리 + 수요 검증 후 검토.

**비용:** LLM 하루 몇 문장(미미) + TTS 0(내장) + 호스팅 거의 0 → **월 몇천 원**으로 매일 갱신 콘텐츠 가능.

**위치:** 습관앱 방향(⑦)의 **킬러 콘텐츠 + 제작 자동화**를 동시에 푸는 아이디어. Phase 2쯤 "오늘의 일본(뉴스 데일리)"로 편입. 단 **저작권 재구성 원칙 + 반자동 검수**는 처음부터 고정.

## ⑨ 상황별 DRILL 카테고리 확장 후보 (2026-07-16)

> ②6순위·④Phase2-4의 "상황별 확장"을 **구체 후보 목록**으로 확정. 현재 한 문장(DRILL)은 회사/일상/여행 3종뿐(N4·N3 각각) → 실제 JLPT 청해·생활 장면을 넓힌다.

### 후보 (우선순위)

| 순위 | 카테고리 | 이유 (JLPT 빈출·실용도) |
|---|---|---|
| ⭐1 | **매장/쇼핑** (店・買い物) | 계산·교환·재고·할인 — N4~N3 최빈출, 여행과 다른 어휘 |
| ⭐2 | **음식점** (レストラン) | 주문·예약·알레르기·계산 — 독립 상황으로 가치 큼 |
| ⭐3 | **학교** (学校) | 수업·과제·시험·동아리 — 청해 지문 단골 |
| 4 | **병원/약국** (病院) | 증상·예약·처방 — 생활 필수 |
| 5 | **교통/길찾기** (交通・道案内) | 환승·지연·분실 — 여행과 분리하면 밀도↑ |
| 6 | **전화/약속** (電話・約束) | 예약변경·부재중 — 청해 대화 핵심 패턴 |

### 실행 메모
- **구조:** 각 카테고리 = 독립 JSON 1개 = 목록 1줄(기존 회사/일상/여행과 동일). 형식은 DRILL(한 문장 듣기) 그대로 — jp/kr/words.
- **자동 인식:** `AssetClipRepository`가 `assets/clips/*.json`을 자동 스캔·id순 정렬 → **파일만 추가하면 코드 수정 없이 목록에 노출.**
- **id 배정(레벨끼리 묶이게):** 현재 N4=1~7, N3=11~17 사용 중. 신규는 **N4: 6·8·9·10 / N3: 16·18·19·20** 등 빈 번호로. (id는 진행위치 저장 키 `pos_{id}`라 한번 정하면 고정)
- **분량:** 카테고리당 100문장(기존 패턴)이 기본이나, 반응 검증용으로 **50문장 착수 → 확장**도 가능.
- **파생 효과:** 랜덤 재생(`randomClip`)이 DRILL 전체 풀에서 뽑으므로, 카테고리가 늘수록 랜덤·"오늘의 10문장"도 자동으로 다양해짐.
- **품질 체크(삽입 시 고정 절차):** id 중복 검사 → jp/kr/words 구조 검증 → 종결·존댓말 톤 확인 → JSON 유효성.
